# 阶段 5A：基于当前综述的二次修改功能

## 1. 目标
在不引入复杂的多轮对话流和历史记录的前提下，为用户提供一个对生成的综述进行“原地打磨”的功能。

## 2. 核心链路设计
1. **输入修改指令**：用户在综述结果下方的文本框内输入指令（如“翻译成英文”、“精简第一段”）。
2. **组装 Prompt**：前端提取 `previousSummaryContent`、`summarySources` 快照和指令发给后端，后端拼装带有严厉规则的 Prompt。
3. **保留溯源能力**：在系统提示词中强制大模型必须使用提供的 `[1]` 等引用标签，防止因为二次修改而丢失溯源依据。
4. **单向流式覆盖**：一旦大模型返回第一个有效 chunk，立刻清空当前的综述，然后启动流式打印覆盖。
5. **异常恢复**：如果在修改过程中报错或被拦截，将恢复修改前备份的旧内容。

## 3. 技术难点与处理
- **防覆盖过早**：点击发送后不能马上清空旧综述（可能网络失败导致全丢）。我们采用了 `isFirstChunk` 标志位，只有真实收到流的第一段内容时才清空。
- **重用 Abort 机制**：共享了原有的 `abortController` 和 `stopGeneration` 方法，让停止按钮能够同时中断“生成”和“修改”两个动作。
- **UI 状态防抖**：添加了 `revising` 状态，输入框和提交按钮在 `loading`、`generating`、`revising` 期间严格 `disabled`。
