# ScholarFlow AI 阶段 1 架构复盘与设计思考

## 一、阶段 1 的核心目标

**这个阶段到底在解决什么问题？**
阶段 1 解决的是“基础设施搭建”和“核心数据流打通”的问题。它验证了项目从文件读取到视图渲染的最底层链路是否可行。

**为什么第一阶段不先做 AI，不先做摘录，而是先做上传 + 基础阅读壳？**
1. **数据依赖**：AI 总结和摘录的前提是“有文本”，文本的前提是“有 PDF 解析结果”。如果没有 PDF 渲染壳，后续的摘录交互（框选、高亮）就失去了宿主，AI 功能也只能是对着空气 mock。
2. **风险前置**：在前端处理 PDF 渲染（使用 pdfjs-dist）和跨组件状态共享（多文档切换）是本项目最大的技术难点和风险点。我们必须在第一阶段把最难啃的骨头（Worker 配置、Canvas 渲染流）啃下来，证明技术路线可行。

## 二、技术栈选择的为什么

**为什么用 Vue 3，而不是 React**
Vue 3 的 Composition API 提供了更自然的心智模型，尤其是没有 React 中令人头疼的闭包陷阱（Stale Closures）和繁琐的依赖数组（dependency arrays）。在处理像 PDF.js 这样大量包含异步操作和外部状态的复杂库时，Vue 3 的 `ref` 和 `watch` 能让逻辑更清晰可控。

**为什么用 Pinia，而不是把状态全写在组件里**
我们的项目是典型的“多视图共享状态”应用（左侧列表、中间阅读器、右侧工作台都需要知道“当前选中的是哪个文档”）。如果写在组件里，就必须通过 `provide/inject` 或者无限的 `props` 传递，导致组件高度耦合。Pinia 提供了一个类型安全的全局状态树，让各个组件可以独立获取所需数据。

**为什么用 Vite，而不是 Webpack**
Vite 基于 ES Modules，冷启动速度是 Webpack 的数十倍。在频繁调试和热更新（HMR）时，Vite 能提供极其流畅的开发体验。同时，Vite 对 TypeScript 和 PostCSS（Tailwind）的内置支持大大减少了工程化配置的成本。

**为什么用 pdfjs-dist，而不是找一个更高层封装的 PDF 组件（如 vue-pdf）**
高层组件虽然开箱即用，但它们**屏蔽了底层的控制权**。我们后续阶段的核心需求是“文本摘录”，这要求我们必须能精确操作 PDF 的 Canvas 层和 TextLayer（文本层），甚至要自己算坐标画高亮框。只有直接使用底层的 `pdfjs-dist`，才能拥有这种细粒度的控制能力。

**为什么现在先不用 Dexie、SSE、IndexedDB 等能力**
工程化讲究“渐进式交付（MVP）”。在第一阶段，如果在核心的 PDF 渲染链路都没跑通的情况下，就去搞 IndexedDB 持久化，一旦出现 Bug，你很难定位是渲染层的问题，还是数据库存取的问题。我们要隔离变量，分步验证。

## 三、三栏布局设计的为什么

**为什么这个项目天然适合三栏布局**
研报阅读是一个典型的“输入 -> 处理 -> 输出”的重度信息工作流。左侧管理输入（文件），中间进行处理（阅读与圈定），右侧展示输出（AI 生成）。这种布局符合人类的阅读习惯和物理世界中“左手翻书，右手做笔记”的直觉。

**左侧、中间、右侧分别承担什么职责**
- **左侧**：资源管理器，负责全局的导航和输入控制。
- **中间**：核心工作台，承担最重的渲染性能压力和交互响应。
- **右侧**：结果面板，负责聚合信息并与外部 API（AI）交互。

**为什么这样拆分比把所有内容堆在一个页面里更合理**
这是“关注点分离”的 UI 体现。如果堆在一起，不仅代码难以维护，DOM 结构也会变得异常臃肿。拆分后，每个区域可以由独立团队/人员开发，甚至可以单独做性能优化（比如只对中间 PDF 区做虚拟滚动）。

**这个布局对后续阶段有哪些铺垫意义**
这种清晰的物理边界，完美对应了我们在 Pinia 中设计的逻辑边界。比如，后续做“对比池”时，我们很清楚数据是从中间区域流向右侧区域的。

## 四、Store 设计的为什么

**为什么 documents 要放全局**
因为左侧（展示列表）、中间（渲染 PDF 内容）、甚至未来右侧（展示引用来源）都需要读取文档数据。

**为什么 activeDocumentId 要放全局**
它是驱动整个应用状态变化的“主引擎”。只要 `activeDocumentId` 变了，中间的阅读器就应该重新渲染，右侧的摘录列表就应该过滤。把它放全局，各个组件只需响应这个单一数据源即可。

**为什么 viewerState 要放全局**
因为可能会有外部组件需要控制阅读器（例如：右侧点击一个 AI 引用，中间的阅读器需要跳转到对应页码）。如果 viewerState 只在 `PdfWorkspace` 内部，外部组件将很难指挥它。

**为什么 uploadError 可以放全局**
虽然目前只有左侧在显示错误，但把错误抽象为全局状态，意味着未来我们可以在顶部弹出一个全局 Toast，或者在右侧工作台中记录系统日志，这增加了扩展性。

**为什么 workbenchMode 要提前占位**
这是架构的“前瞻性”。提前定义好 `excerpt`（摘录模式）和 `summary`（综述模式），能指引我们接下来的状态流转设计。

**哪些状态不应该放进 Pinia，为什么**
例如 `PdfWorkspace.vue` 中的 `canvasRef`、加载时的 `loading` 动画状态、或者是用户鼠标在屏幕上的当前坐标。这些属于**纯粹的 UI 表现状态**或**局部计算状态**，把它们放进 Pinia 会导致 Store 臃肿，且违反了单一数据源原则。

## 五、Composables 设计的为什么

**为什么 PDF 加载和渲染逻辑不直接写在 PdfWorkspace.vue 里**
如果在 Vue 组件里写这几百行调用 `pdfjs-dist` API 的代码，组件会变得巨大无比，难以阅读。Vue 组件的职责应该是“描述 UI 结构”，而不是“实现复杂的业务算法”。

**为什么这种 DOM / canvas / pdfjs 相关逻辑适合抽成 composable**
Composable（如 `usePdf`）可以封装有状态的逻辑（`loading`, `error`, `pdfDoc`）。它像一个黑盒，组件只需要喂给它一个 File，它就吐出渲染方法和状态。这不仅让组件干净，而且如果未来我们需要在另一个页面（比如全屏阅读模式）也渲染 PDF，直接复用这个 composable 即可。

**composable 和 store 的边界是什么**
Store（Pinia）负责跨组件共享的**业务数据**（应用层面的事实，如“有几篇文档”）。Composable 负责**局部状态和逻辑复用**（机制层面的实现，如“如何把 PDF 画在 Canvas 上”）。

**composable 和 component 的边界是什么**
Composable 是没有 UI 的（Headless）。它只产出数据和方法。Component 负责消费这些数据，并把它们绑定到具体的 DOM（如 `<canvas>`、`<div>`）上。

## 六、组件拆分的为什么

**为什么上传区、阅读区、工作区要拆成独立组件**
降低耦合度。左侧代码哪怕完全写废了，也不会导致中间 PDF 渲染崩溃。同时方便在不同文件中单独管理各自特有的局部状态和样式。

**为什么 PdfWorkspace 只负责“接 store + 调 composable + 展示状态”**
这遵循了“展示型组件与容器型组件分离”的设计模式。PdfWorkspace 是一个典型的容器组件，它像一个胶水层，把全局数据（Store）和底层能力（Composable）粘合在一起，最后传递给具体的 DOM 元素去展示。

**为什么组件不应该既管全局状态又管底层渲染实现**
违反“单一职责原则”。如果一个组件又要管“别人传了什么文件”，又要管“PDF.js 的 Worker 怎么加载”，还要管“Canvas 像素怎么画”，那这个文件最终会变成一座不可维护的“屎山”。

## 七、当前阶段的架构优点

- **可维护性**：视图（Components）、状态（Stores）、底层能力（Composables）三者完全解耦。
- **可扩展性**：如果明天产品经理说要增加一个“Word 文档阅读”功能，我们只需新建一个 `useWord.ts` composable，Store 和布局基本不用动。
- **后续接 AI 的便利性**：三栏布局和状态设计已经把“操作源”和“数据源”理清，AI 模块只需作为新的消费者订阅 Store 中的摘录数据即可。
- **面试时可讲述性**：这套架构完美体现了 Vue 3 的精髓。面试时可以清晰地讲出“我如何用 Pinia 管理领域模型，用 Composable 隔离外部依赖（pdf.js），用 Tailwind 提高 UI 效率”。

## 八、当前阶段的局限和后续演进

**当前阶段还缺什么**
- 缺少 TextLayer（文本层），导致 PDF 无法选中文本，这是做摘录的致命阻碍。
- 缺少分页和缩放控制。

**哪些设计只是为了阶段 1 的 MVP**
- `usePdf` 里硬编码了只渲染 `pageNumber: 1`。
- `documents` 数组只存在内存里，一刷新就没了。

**后续阶段最自然的演进方向是什么**
1. 完善 `usePdf`，引入 TextLayer 渲染，打通文字选中能力（DOM Selection API）。
2. 在 UI 上添加翻页和缩放控制器。
3. 引入 Dexie，把 `documents` 从内存同步到 IndexedDB。

## 九、面试表达版

“在做 ScholarFlow AI 的第一阶段时，我的核心目标是**打通基础设施和数据流转**。

在技术选型上，我果断选择了 **Vue 3 的 Composition API** 配合 **Pinia**。因为我们这是一个典型的‘多视图共享状态’的工作站应用——左侧上传列表、中间阅读器、右侧 AI 面板都需要实时同步‘当前在看哪篇文档’这个核心状态。用 Pinia 可以很干净地把 `activeDocumentId` 等领域数据抽离出来。

至于 PDF 渲染，我没有用现成的高级组件，而是直接集成了底层的 **pdfjs-dist**。因为我考虑到后续的核心功能是‘文本高亮和摘录’，这必须要求我能精确控制 Canvas 和 TextLayer 的 DOM 结构。

在工程化落地上，我非常注重**关注点分离**。我把 pdfjs 原生的加载、解析、Canvas 渲染这套极度复杂的逻辑，完全封装进了一个叫 `usePdfRenderer` 的 **Composable** 里。这让我的 UI 组件变得极其薄，只负责从 Pinia 拿数据，传给 Composable，然后把结果渲染到屏幕上。

这种设计的好处是，现在我的代码结构非常清晰，即使接下来第二阶段要加上复杂的文本选中和框选逻辑，我也只需要在 Composable 层进行迭代，完全不会污染 UI 层，保证了项目极强的可扩展性。”
